# 通用设计文档规则

版本：1.0

适用范围：所有项目的设计文档、系统分段、设计冻结文档、生产交接文档。

本文从步骤 2 的 Agent 契约、全局冻结标准、质量闸门和单系统标准中抽取通用规则。它只定义设计文档应满足的结构、事实边界和质量要求，不定义任何具体项目的实体、资源、阵营、地点、剧情、数值或专有术语。

现有步骤 2 文档继续作为流水线执行细则使用；本文是更上层的通用写作与评审规则，不改变现有流程。

## 1. 文档目标

一份合格的设计文档不是创意合集，也不是讨论记录，而是可被后续策划、程序、美术、关卡、QA 和生产管理消费的设计契约。

设计文档必须回答：

1. 玩家为什么要参与核心循环。
2. 玩家做什么决策。
3. 玩家为决策支付什么代价或承担什么风险。
4. 系统如何反馈结果。
5. 结果如何改变后续压力、资源、状态或策略空间。
6. 哪些事实已经冻结，哪些风险尚未关闭。
7. 下游生产应该实现、表现、验证什么。

设计文档不得承担：

1. 保存头脑风暴备选方案。
2. 隐藏未解决冲突。
3. 用模糊描述替代规则。
4. 用实现细节反向定义设计事实。
5. 把内部治理字段、提示词或扫描器数据暴露给生产正文。

## 2. 通用化边界

通用规则只能描述结构、职责、检查方法和质量门槛。

禁止在通用规则中写入：

1. 项目专有实体名。
2. 项目专有资源名。
3. 项目专有阵营、角色、地点或组织。
4. 项目专有剧情事实。
5. 项目专有数值阈值。
6. 只适用于单一题材或单一机制的规则。

示例必须使用抽象占位，例如：

```text
[玩家可控对象]
[关键资源]
[风险状态]
[高收益行动]
[长期后果]
```

占位示例只约束结构，不是项目事实。项目事实只能来自用户输入、上游设计方向、已批准分段、治理裁决或事实契约。

## 3. 权威顺序

当多个来源互相冲突时，按以下顺序裁定事实：

| 优先级 | 来源 | 权威范围 |
|---:|---|---|
| 1 | 人类输入和已批准方向 | 产品意图、体验边界、不可违背的项目目标 |
| 2 | 治理裁决 | 失败后对冲突或阻断问题的唯一规则裁定 |
| 3 | 事实契约 | 正式术语、阈值、资源职责、不变量 |
| 4 | 归属分段 | 本领域事实的展开和细节 |
| 5 | 最终文档 | 聚合事实，不创造新事实 |

最终文档只能汇总、整理和表达已存在的事实。若最终文档推翻上游方向、治理裁决、事实契约或归属分段，最终文档错误。

质量评审只判断是否可冻结，不新增设计事实。质量失败后，必须由治理裁决或归属分段修改事实，不能由最终文档自行修复。

## 4. 一事实一归属

每个正式设计事实只能有一个 owner。

| 事实类型 | 常见归属 |
|---|---|
| 核心体验、玩家动词、核心循环、推进压力 | gameplay |
| 资源、产出、消耗、仓储、衰减、转化 | economy |
| 活跃漏洞、利用路线、负向测试 | exploit |
| 点击流、信息层级、警告、认知负担 | ux |
| 状态机、事件顺序、持久化、确定性计算 | technical |
| 世界因果、组织关系、规则解释 | world_rules |
| 视觉语言、美术红线、可读性要求 | visual_rules |
| 跨分段冲突和证据 | consistency |
| 聚合后的生产正文 | final_doc |

非归属分段可以引用事实，但不得重定义事实。

如果一个系统必须跨多个分段才能成立，文档必须拆分：

1. 本分段只定义自己拥有的事实。
2. 其它事实写入引用事实或依赖矩阵。
3. 跨分段共用的规则写入不变量或事实契约。
4. 任何分段不得用不同术语重命名同一事实。

## 5. 设计内核

任何设计文档都应先形成设计内核。设计内核是全部系统共同服从的最小事实集合。

必需字段：

| 字段 | 要求 |
|---|---|
| `core_promise` | 一句话说明玩家最终体验。 |
| `primary_loop` | 闭环：动机 -> 行动 -> 代价或风险 -> 反馈 -> 状态变化 -> 新动机。 |
| `main_pressure` | 阻止玩家无后果停滞的主要压力。 |
| `main_reward` | 让玩家愿意承担风险的主要收益。 |
| `progression_axis` | 复杂度、压力或策略空间增长的主轴。 |
| `irreversible_costs` | 无法被重置、刷取或转化为收益的代价。 |
| `failure_state` | 玩家拒绝或失败时发生的结果。 |
| `world_or_context_constraint` | 解释规则为何成立的世界、题材或产品约束。 |

设计内核不得包含：

1. 数据库字段。
2. 类名、接口名或引擎对象。
3. 提示词。
4. 质量闸门原始意见。
5. 扫描器字段。
6. 仍未裁定的备选方案。

## 6. 系统分段规则

每个系统分段必须以玩家可感知结果为中心。实现层可以有不可见账本，但设计文档必须说明玩家能看到、预判、利用、失去或决策的效果。

系统分段最低结构：

```text
系统 ID：
正式名称：
归属分段：
拥有事实：
引用事实：
玩家承诺：
设计目的：
进入条件：
退出条件：
玩家动作：
系统反馈：
状态变量：
资源产出：
资源消耗：
存储或持久化：
失败状态：
不可逆后果：
上游依赖：
下游影响：
漏洞风险：
UX 需求：
技术影响：
验证用例：
```

若字段不适用，写 `无`，并说明原因和下游影响；不得直接省略。

### 6.1 玩家承诺

玩家承诺必须说明玩家做什么决策，以及该决策为什么重要。

不合格：

```text
该系统增加深度。
```

合格：

```text
玩家决定是否承担更高失败风险执行高收益行动，以换取能改变后续策略空间的关键收益。
```

### 6.2 闭环

每个核心系统必须形成闭环：

```text
动机 -> 玩家行动 -> 代价或风险 -> 反馈 -> 状态变化 -> 新动机
```

如果系统无法回到玩家动机，它只能作为辅助系统，不能作为核心推进门槛。

### 6.3 进入与退出

进入条件必须说明：

1. 谁触发系统。
2. 触发是可选、强制还是自动。
3. 首次触发后是否锁定。
4. 触发失败时发生什么。

退出条件必须说明：

1. 成功退出。
2. 失败退出。
3. 中断退出。
4. 超时退出。
5. 退出后状态是重置、保留、锁定还是变异。

### 6.4 状态变量

状态变量格式：

```text
[变量名]：
- 含义：
- 归属：
- 被谁改变：
- 被谁读取：
- 重置规则：
- 锁定规则：
- 玩家可见反馈：
```

影响推进、经济、风险、权限或长期后果的变量，必须进入事实契约或被不变量引用。

同名变量不得在多个系统中拥有不同归属。若多个系统读写同一变量，必须明确唯一 owner 和所有 reader/writer。

## 7. 全局不变量

不变量是跨所有分段都必须保持为真的规则。它必须可验证、可引用、可阻断。

常见不变量类型：

1. 推进不变量：什么不能被永久拖延。
2. 经济不变量：什么不能被无限产出。
3. 仓储不变量：什么不能无代价保存。
4. 压力不变量：什么不能通过布局、时序、分组或状态投机规避。
5. 后果不变量：什么不能通过后续操作抹除。
6. 失败不变量：什么不能被转化为收益路线。
7. 术语不变量：哪些名称是正式名称。

不变量格式：

```text
INV-[领域]-[编号]
规则：[一句话]
理由：[它保护了什么核心体验]
适用分段：[分段列表]
禁止绕过：[玩家不应成功的策略]
验证方式：[质量评审如何检查]
```

同类硬阻断反复出现时，必须升级为不变量。只改措辞不算修复。

## 8. 资源与经济规则

每种资源都必须进入资源总账。资源包括货币、点数、材料、状态、权限、关系、声望、风险值、控制力和其它可被获得、保存、消耗或转化的设计对象。

资源总账格式：

```text
资源：
- 正式名称：
- 类型：
- 产出：
- 消耗：
- 存储：
- 衰减或过期：
- 溢出处理：
- 可转化为：
- 禁止用途：
- 主要玩家决策：
- 滥用风险：
```

资源规则：

1. 不存在只有产出没有消耗的资源。
2. 不存在只有消耗没有产出或获取方式的资源。
3. 不存在无人消费、无人感知的资源。
4. 无限仓储必须被核心循环需要且经过滥用验证，否则不允许冻结。
5. 临时奖励必须声明持续时间、过期方式、能否修复以及反刷取机制。
6. 抽象资源必须有独立正式职责，不能静默替代其它资源。
7. 资源转化必须说明转化方向、损耗、限制和禁止路线。

## 9. 阶段、推进与不可逆后果

阶段必须是状态，不是氛围描述。

阶段格式：

```text
阶段：
- 进入条件：
- 首次达成后是否锁定：
- 退出条件：
- 强制推进计时器：
- 玩家目标：
- 新增压力：
- 新增资源：
- 新增失败路线：
- 反停滞规则：
- 回退规则：
```

推进规则：

1. 若某阈值首次达到后不可逆，必须写明锁定。
2. 若玩家可长期低于某阈值，必须定义强制事件。
3. 若玩家可通过微操对象数量、库存、评分、关系或状态规避推进，必须定义微操结果。
4. 计时器必须说明启动、暂停、重置和锁定规则。
5. 回退必须说明是允许策略、失败惩罚还是被禁止路线。

任何用于强制推进的惩罚，都不能只依赖玩家可控分母。

不合格：

```text
惩罚 = 当前玩家可控对象数量的 50%
```

合格：

```text
惩罚 = max(相对惩罚, 绝对下限) + 不足额外后果
```

绝对后果必须定义最低损失、低于下限时的不足惩罚、长期代价和负向测试。

## 10. 依赖矩阵

冻结或交接前必须存在依赖矩阵。

最低列：

| 系统 | 上游输入 | 下游输出 | 共享状态 | 失败影响 | 归属分段 |
|---|---|---|---|---|---|

检查项：

1. 不存在循环解锁死锁。
2. 不存在消耗无生产者的状态或资源。
3. 不存在产出无人消费、无人感知的死端系统。
4. 不存在多个系统同时拥有同一正式事实。
5. 子系统不得越权改变其它分段拥有的事实。

依赖必须点名具体状态、资源、规则或不变量 ID。不得只写“依赖上游设计”“由其它系统提供”等泛称。

## 11. 漏洞与边界情况

每个系统至少检查以下滥用路径：

1. 无限产出。
2. 无限仓储或保鲜。
3. 免费取消或退款循环。
4. 故意失败获利。
5. 阈值反复横跳。
6. 对象数量操控。
7. 布局、位置或时序规避。
8. 归属、分组、等级或权限重分配规避。
9. 计时器重置滥用。
10. 已修复漏洞回归。

检查格式：

```text
检查类别：
滥用路线：
玩家收益动机：
被破坏的不变量：
阻断规则：
残余风险：
状态：active / fixed / obsolete / accepted-risk
```

状态规则：

1. `active` 且高严重度或形成最优路线的漏洞是冻结阻断。
2. `fixed` 必须有阻断规则和负向测试。
3. `obsolete` 只用于底层系统已移除或替换。
4. `accepted-risk` 必须有非阻断理由、归属分段和后续验证方式。
5. `accepted-risk` 不得用于会破坏核心承诺或形成支配性策略的漏洞。

负向测试格式：

```text
漏洞路线：
正常路线：
修复后漏洞路线结果：
正常路线结果：
为什么漏洞路线更差：
```

## 12. UX 契约

任何改变玩家状态的系统都必须定义：

1. 行动前玩家必须知道什么。
2. 行动中玩家看到什么。
3. 成功如何确认。
4. 失败如何确认。
5. 不可逆后果如何警告。
6. 哪些信息可以延迟展示。
7. 哪些操作会造成认知负担或误触风险。

UX 契约不得定义最终 UI 美术、具体控件实现、文件名或代码结构。

## 13. 技术契约

技术契约只描述可实现性边界，不写代码。

允许包含：

1. 必需状态机。
2. 必需事件顺序。
3. 必需持久化。
4. 必需确定性计算。
5. 必需跨系统通知。
6. 必需追踪的失败状态。

禁止包含：

1. 数据库字段名。
2. 网络协议细节。
3. 尚未标准化的类名。
4. 引擎对象引用。
5. Debug、GM 或测试命令作为设计事实。

技术契约不能用实现限制推翻已批准设计事实。若实现风险改变设计边界，必须回到归属分段或治理裁决。

## 14. 世界、叙事与视觉规则

世界、叙事和视觉规则的职责是支撑玩法可信性、可读性和生产一致性，不得新增玩法机制或经济规则。

世界规则应说明：

1. 核心前提。
2. 规则为何成立。
3. 组织、关系、地点或实体的因果约束。
4. 资源来源和风险逻辑。
5. 哪些叙事解释会破坏玩法规则。

视觉规则应说明：

1. 造型语言。
2. 材质语言。
3. 色彩与光照规则。
4. 分组、等级、风险或状态的可读性。
5. 必须表现的不变量。
6. 禁止描绘的视觉矛盾。

世界和视觉分段可以解释和表现已存在规则，但不得越权创建新的核心循环、资源消耗、状态机或数值阈值。

## 15. 质量评审

质量评审只判断是否可冻结或可交接，不负责直接写修复方案。

质量评审必须检查：

1. 核心承诺是否清晰。
2. 核心循环是否闭合。
3. 各分段事实是否一致。
4. 资源、经济和漏洞风险是否受控。
5. 玩家是否能理解、预判并操作。
6. 技术状态是否无明显歧义。
7. 世界解释与视觉规则是否支撑玩法。
8. 最终文档是否干净、面向生产。

每个低分、阻断或返工要求必须附带证据：

```text
BLOCKER-ID：
- 问题：
- 证据分段：
- 证据声明：
- 被破坏的不变量：
- 受影响分段：
- 必需返工类型：
```

禁止输出“还需优化”“不够完善”“体验不足”这类无证据阻断。阻断必须指出具体事实、具体分段和具体被破坏的规则。

## 16. 最终文档边界

最终设计文档是面向生产的设计事实文档。

允许包含：

1. 正式设计规则。
2. 设计内核。
3. 系统摘要。
4. 资源总账。
5. 阶段推进状态。
6. 依赖矩阵。
7. 世界规则。
8. 视觉规则摘要。
9. UX 与技术影响。
10. 非阻断风险。

禁止包含：

1. 内部治理裁决原始 JSON。
2. 事实契约原始 JSON。
3. 质量闸门原始输出。
4. 提示词或 Agent 指令。
5. `forbidden_terms`。
6. `forbidden_claims`。
7. `required_terms`。
8. `required_claims`。
9. `canonical_terms`。
10. `affected_sections`。
11. 禁止术语清单。
12. 活跃漏洞复现正文，除非文档明确是失败审查产物。
13. 已修复漏洞作为当前活跃问题重复列出。
14. 在质量通过前自称“可冻结”。

如果需要表达禁止概念，必须写成正向规则。

不合格：

```json
"forbidden_terms": ["旧名称"]
```

合格：

```text
该资源的正式名称是[名称]，全文只使用该名称。
```

## 17. 语言与格式规则

设计正文必须使用可执行、可验证、可交接的表达。

禁止使用：

1. “可能”。
2. “大概”。
3. “视情况而定”。
4. “玩家自然会理解”。
5. “系统自行处理”。
6. “后续优化”。
7. “已经稳定”。
8. “可以冻结”，除非质量评审已经通过。

如果确实存在条件分支，必须写清：

1. 条件是什么。
2. 谁触发。
3. 系统如何判断。
4. 成功结果是什么。
5. 失败结果是什么。
6. 下游如何验证。

未定义缩写不得出现。首次出现的术语必须给出正式名称和职责。

## 19. 合格清单

设计文档通过前必须逐条满足：

1. 核心承诺清晰，并能被核心循环验证。
2. 核心循环至少可追踪三轮。
3. 每个主要系统都有玩家可感知意义。
4. 每个系统都有进入、退出、失败和中断规则。
5. 每个资源都有产出、消耗、存储、溢出或衰减。
6. 每个不可逆后果都说明为何不可逆。
7. 每个跨系统依赖都点名具体状态、资源、规则或不变量。
8. 每个活跃漏洞都有状态、归属、阻断规则或阻断结论。
9. 高严重度漏洞必须有负向测试证明关闭，否则不得冻结。
10. 所有正式术语一致。
11. 没有多个分段同时拥有同一事实。
12. 世界、视觉、UX 和技术分段没有越权新增玩法事实。
13. 最终文档没有内部治理元数据、提示词或扫描器字段。
14. 最终文档没有创造来源分段不存在的新事实。
15. 所有 `无` 或“不适用”都有原因和下游影响。
16. 所有非阻断风险都有归属分段和后续验证方式。

任一项失败，文档不得作为冻结设计或生产交接依据。
