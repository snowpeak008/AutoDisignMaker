# 存档架构对比：旧 vs 新

## 架构演进概览

```
【旧架构】分散存储 → 【新架构】统一执行对象存储
```

---

## 一、数据流对比

### 旧架构：多路径分散存储

```
┌─────────────────────────────────────────────────────────────────┐
│ 用户操作                                                         │
└─────────────────────────────────────────────────────────────────┘
         │
         ├─→ 设计工作台保存 → 文件对话框 → projects/test__001.json
         │                                    ↓
         │                             ❌ 不在存档范围
         │                             ❌ 无版本历史
         │                             ❌ 手动管理
         │
         ├─→ 设计工作台导出 → 文件对话框 → sandbox/workspace/exports/*.json
         │                                    ↓
         │                             ❌ 不在存档范围
         │                             ❌ 丢失关联
         │
         └─→ 流水线执行 → saves/{save_id}/workspace/
                              ├─ sandbox/source_artifacts/
                              ├─ sandbox/outputs/artifacts/
                              └─ gate_log.yaml
                              ❌ 不包含设计项目
                              ❌ 不包含用户导出
```

**问题**：
- ❌ 数据散落在 3 个不同位置
- ❌ 设计项目与流水线存档分离
- ❌ 切换存档不会切换设计项目
- ❌ 无法追溯设计项目的历史版本
- ❌ 手动管理文件，容易丢失

---

### 新架构：统一执行对象存储

```
┌─────────────────────────────────────────────────────────────────┐
│ 用户操作                                                         │
└─────────────────────────────────────────────────────────────────┘
         │
         ├─→ 设计工作台保存 ─┐
         │                  │
         ├─→ 设计工作台导出 ─┤
         │                  │
         └─→ 流水线执行 ─────┤
                            ↓
                   ┌────────────────────┐
                   │ ExecutionObjectStore │
                   └────────────────────┘
                            ↓
         saves/{save_id}/workspace/outputs/execution_objects/
                     execution_objects.json
                            ↓
                   ┌────────────────────┐
                   │  统一存储的对象类型  │
                   ├────────────────────┤
                   │ • design_project   │ ← 设计项目
                   │ • user_artifact    │ ← 用户导出
                   │ • workspace_snapshot│ ← 工作区快照
                   │ • program_task     │ ← 程序任务（已有）
                   │ • art_task         │ ← 美术任务（已有）
                   │ • rollback_plan    │ ← 回滚计划（已有）
                   └────────────────────┘
                            ↓
                   ✅ 在存档范围内
                   ✅ 完整版本历史
                   ✅ 自动管理
                   ✅ 统一查询接口
```

**优势**：
- ✅ 所有数据在一个位置
- ✅ 设计项目与流水线存档完全集成
- ✅ 切换存档自动切换设计项目
- ✅ 完整的版本历史和审计轨迹
- ✅ 系统自动管理，不会丢失

## 二、目录结构对比

### 旧架构目录结构

```
AutoDesignMaker/
├── projects/                      【问题1：独立于存档系统】
│   └── test__001.json             ❌ 不会被流水线存档备份
│
├── sandbox/
│   ├── source_artifacts/          ✅ 在存档范围
│   ├── outputs/                   ✅ 在存档范围
│   └── workspace/                 【问题2：不在存档范围】
│       ├── projects/              ❌ 不会被备份
│       │   └── another.json
│       └── exports/               ❌ 不会被备份
│           └── export_001.json
│
└── saves/
    └── {save_id}/
        ├── save_manifest.json
        ├── timeline.jsonl
        └── workspace/
            ├── sandbox/
            │   ├── source_artifacts/    ✅ 已备份
            │   └── outputs/             ✅ 已备份
            └── gate_log.yaml            ✅ 已备份
            ❌ 缺少：设计项目
            ❌ 缺少：用户导出
```

**ACTIVE_DIRS（旧）**：
```python
ACTIVE_DIRS = (
    "sandbox/source_artifacts",
    "sandbox/outputs",
    # ❌ 缺少 sandbox/workspace
)
```

---

### 新架构目录结构

```
AutoDesignMaker/
├── projects/                      【可选：保留备份】
│   └── test__001.json.bak         💾 迁移后的备份
│
├── sandbox/
│   ├── source_artifacts/          ✅ 在存档范围
│   └── outputs/                   ✅ 在存档范围
│       ├── artifacts/
│       └── execution_objects/     ✅ 【新增】执行对象存储
│           └── execution_objects.json  ← 所有数据的单一真相源
│
└── saves/
    └── {save_id}/
        ├── save_manifest.json
        ├── timeline.jsonl
        └── workspace/
            ├── sandbox/
            │   ├── source_artifacts/
            │   └── outputs/
            │       ├── artifacts/
            │       └── execution_objects/  ✅ 包含所有执行对象
            │           └── execution_objects.json
            │               ├─ design_project     (设计项目)
            │               ├─ user_artifact      (用户导出)
            │               ├─ workspace_snapshot (工作区快照)
            │               └─ ...
            └── gate_log.yaml
```

**ACTIVE_DIRS（新）**：
```python
ACTIVE_DIRS = (
    "sandbox/source_artifacts",
    "sandbox/outputs",  # ← 包含 execution_objects/
    # ✅ execution_objects 自动包含在 outputs 中
)

EMPTY_DIRS = (
    # ... 现有目录
    "sandbox/outputs/execution_objects",  # 新增
)
```

## 三、操作流程对比

### 操作1：保存设计项目

#### 旧方式
```
1. 用户点击"保存"按钮
2. 弹出文件对话框
3. 用户选择路径和文件名
4. 写入 JSON 文件到磁盘
5. 完成（文件独立存在，不在存档中）
```

**用户体验**：
- 😕 需要选择保存位置
- 😕 需要记住文件名
- 😕 可能保存到错误位置
- 😕 无法追溯历史

#### 新方式
```
1. 用户点击"保存"按钮
2. 直接保存到执行对象存储（无对话框）
3. 自动生成版本ID（如 EO-000001）
4. 自动触发存档同步
5. 完成（已在存档范围内）
```

**用户体验**：
- 😊 一键保存，无需选择路径
- 😊 自动版本管理
- 😊 自动备份
- 😊 完整历史记录

---

### 操作2：打开设计项目

#### 旧方式
```
1. 用户点击"打开"按钮
2. 弹出文件对话框
3. 用户浏览文件系统找文件
4. 选择 JSON 文件
5. 加载项目
```

**用户体验**：
- 😕 需要记住文件位置
- 😕 可能找错文件
- 😕 无法看到版本历史

#### 新方式
```
1. 用户点击"打开"按钮
2. 显示版本列表对话框
   ┌────────────────────────────────────────┐
   │ 选择设计项目版本                        │
   ├────────────────────────────────────────┤
   │ EO-000003 | test__001 | 2026-06-20 15:30 | verified │
   │ EO-000002 | test__001 | 2026-06-20 14:20 | verified │
   │ EO-000001 | test__001 | 2026-06-20 10:00 | verified │
   └────────────────────────────────────────┘
3. 用户选择版本
4. 加载项目
```

**用户体验**：
- 😊 清晰的版本列表
- 😊 可以选择任意历史版本
- 😊 显示保存时间和状态
- 😊 支持版本比较（未来功能）

---

### 操作3：切换存档

#### 旧方式
```
1. 用户切换到存档 A
2. 流水线数据恢复到存档 A 的状态
3. 设计项目仍然是之前的（未切换）❌
4. 用户需要手动打开对应的设计项目文件
```

**问题**：
- 😕 设计项目与流水线不同步
- 😕 容易混淆状态
- 😕 需要手动管理

#### 新方式
```
1. 用户切换到存档 A
2. 流水线数据恢复到存档 A 的状态
3. 设计项目自动恢复到存档 A 的状态 ✅
4. 所有数据完全同步
```

**优势**：
- 😊 设计项目与流水线同步
- 😊 状态一致，不会混淆
- 😊 完全自动化

## 四、数据完整性对比

### 旧架构：数据孤岛

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ 设计项目     │     │ 用户导出     │     │ 流水线制品   │
│ projects/   │     │ exports/    │     │ artifacts/  │
└─────────────┘     └─────────────┘     └─────────────┘
      ❌              ❌              ✅
   无关联           无关联          已关联

无法回答的问题：
• 这个导出来自哪个设计项目版本？
• 这个设计项目对应哪个流水线状态？
• 如何回滚到某个一致性状态？
```

---

### 新架构：完整关联

```
┌─────────────────────────────────────────────────┐
│         ExecutionObjectStore                    │
│                                                 │
│  ┌──────────────┐   references   ┌───────────┐ │
│  │ design_project├──────────────→│user_artifact│
│  │ EO-000001    │                │ EO-000002  │ │
│  └──────┬───────┘                └───────────┘ │
│         │                                       │
│         │ linked_to                             │
│         ↓                                       │
│  ┌──────────────┐   triggers   ┌─────────────┐ │
│  │program_task  ├──────────────→│ rollback_plan│
│  │ EO-000003    │                │ EO-000004  │ │
│  └──────────────┘                └─────────────┘ │
└─────────────────────────────────────────────────┘

可以回答的问题：
• ✅ 这个导出来自哪个设计项目？ → related_facts.source_project_id
• ✅ 这个设计项目什么时候保存的？ → state_history
• ✅ 如何回滚到某个版本？ → restore_design_project_version(id)
• ✅ 哪些任务依赖这个设计？ → 查询 write_scope
```

## 五、技术指标对比

| 指标 | 旧架构 | 新架构 | 改进 |
|------|--------|--------|------|
| **存储位置** | 3+ 个目录 | 1 个文件 | 🟢 统一 |
| **版本历史** | 无 | 完整 state_history | 🟢 可追溯 |
| **数据关联** | 无 | 完整引用链 | 🟢 可查询 |
| **备份策略** | 手动 | 自动（随存档） | 🟢 自动化 |
| **切换存档** | 部分切换 | 完全切换 | 🟢 一致性 |
| **Schema 验证** | 无 | 自动验证 | 🟢 数据安全 |
| **查询接口** | 文件系统 | 统一 API | 🟢 易用性 |
| **性能** | 散列 I/O | 单文件读写 | 🟡 取决于大小 |
| **可扩展性** | 低 | 高（类型注册） | 🟢 易扩展 |
