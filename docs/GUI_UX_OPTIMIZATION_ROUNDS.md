# 程序自动开发流程工具 UI/UE 五轮自我优化记录

本文记录 `DevFlow.exe` 的五轮界面和体验优化结果。

## 第 1 轮：信息架构审查

发现的问题：

- 顶部按钮过多，操作者需要自己判断命令关系。
- 阶段列表只有底层状态，没有直接说明下一步动作。
- 人工输入、运行、验收之间的路径不够明确。
- 日志和验收都是长文本，缺少完成后的导向。

优化方向：

- 把“命令集合”改成“工作台”。
- 让界面先回答三个问题：当前项目状态如何、当前阶段是什么、下一步做什么。

## 第 2 轮：主界面重设计

已实现：

- 顶部新增项目健康度：通过、缺源资料、待运行、需处理、源资料数量。
- 操作区按用途分组：阶段运行、检查验收、打开目录。
- 右侧新增当前阶段摘要：处理状态、阶段校验、源资料、缺失数量。
- 阶段列表新增处理状态，并用颜色区分已通过、缺源资料、待运行、需处理。

体验变化：

- 操作者进入界面后不需要先读日志，就能知道项目推进状态。
- 阶段列表不再只是技术报告索引，而是流程导航。

## 第 3 轮：人工输入体验强化

已实现：

- 每个阶段自动载入独立草稿，切换阶段不会误带上一阶段内容。
- 输入区显示草稿状态、输入字数、确认进度。
- 确认清单支持全部确认和清空确认。
- 第 0 阶段空输入会被拦截。
- 其他阶段空输入、未勾选确认项、未确认通过后直接运行，都会二次确认。
- 修复“提交失败仍继续运行”的流程问题。

体验变化：

- 操作者更容易知道当前输入是否已经保存、是否满足进入下一步的条件。
- 阶段之间的人工内容边界更清晰。

## 第 4 轮：验收和运行反馈优化

已实现：

- 运行任务使用进度条表示忙碌状态。
- 运行中禁用会造成冲突的操作按钮。
- 日志页对标题、成功输出、失败输出做基础颜色区分。
- 运行成功后自动切到验收结果页。
- 运行失败后停留在日志页，便于排错。
- 总体验收显示通过、缺源资料、待运行和需处理阶段数量。

体验变化：

- 操作者能明确判断任务是否仍在运行。
- 运行完成后界面自动带到最有用的信息位置。

## 第 5 轮：验证和交付

验收标准：

```text
GUI 能正常实例化
中文控件文案不暴露 source_artifacts/outputs 等内部入口
DevFlow.exe --self-test 通过
compileall 通过
根目录 exe 重新打包并替换
```

保留边界：

- 内部命令、目录前缀、JSON 字段仍使用英文稳定 key。
- 界面表现、按钮、阶段状态、资料包含义使用中文。
- 原始运行日志仍保留底层输出，便于排错和追踪。
