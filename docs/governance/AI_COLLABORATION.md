## 二、任务生命周期状态机

### 2.1 状态定义

AI 接手任务后，按以下状态机运行。允许根据条件回跳至前序状态。
RECEIVED
↓
ANALYZING (确定档位、加载最小协议)
↓
RISK_SCAN (扫描项目语义包中的危险实体与禁止依赖)
↓
PLANNING (生成实施路线)
↓
IMPLEMENTING (执行修改，每文件修改后评估是否需要回跳)
↓
VERIFYING (编译、运行时、状态验证)
↓
ENTROPY_CHECK (架构熵检测)
↓
COMPLETING (文档同步、产出清单)

text

### 2.2 各状态激活的规则

| 状态 | 激活的治理规则 | 来源 |
|------|---------------|------|
| ANALYZING | 档位判定、任务类型识别、动态加载矩阵 | 本文件第三、八节 |
| RISK_SCAN | 项目语义包（Critical Entity、Forbidden Dep、T1热路径、Failure Memory） | 外部语义包 |
| PLANNING | P2 架构一致性规则、跨模块通信围栏 | 工程基准第四节 |
| IMPLEMENTING | 全量 P0-P1 工程规则（性能红线、防御性编程、序列化等） | 工程基准第三至七节 |
| VERIFYING | 防御性检查清单、编译零容忍、测试要求、验证契约 | 工程基准第六、十四节 |
| ENTROPY_CHECK | 熵增检测阈值、补丁计数器、架构漂移 | 本文件第六节 |
| COMPLETING | 文档同步、产出清单、ADR 判定、治理行为摘要 | 本文件第四、七节 |

### 2.3 回跳条件

| 当前状态 | 触发条件 | 回跳到 |
|----------|---------|--------|
| IMPLEMENTING | 新增修改文件触及 `CRITICAL` 实体 | RISK_SCAN |
| IMPLEMENTING | 新增 `using` 触及禁止依赖表 | RISK_SCAN（并立即阻止该引用） |
| IMPLEMENTING | 修改序列化/存档核心结构 | RISK_SCAN（自动升级风险） |
| VERIFYING | 编译失败且原因不在当前文件 | ANALYZING |
| ENTROPY_CHECK | 触发 Entropy Abort | RISK_REVIEW（暂停并等待人类决策） |
| 任何状态 | 发现此前未识别的 Critical Entity | RISK_SCAN |

## 三、动态档位系统

### 3.1 档位定义

| 档位 | 适用场景 | 执行策略 |
|------|----------|---------|
| T1 微任务 | 单文件修改、局部 Bug 修复、独立简易功能 | 无需完整计划，信任 AI 经验直接编码 |
| T2 中任务 | 常规业务模块、中等规模迭代 | 提供精简架构指引，AI 可补充工具类 |
| T3 大任务 | 核心底层重构、全局接口动刀、跨模块影响 | 启动最高安全模式，完整架构计划与防错自省 |

### 3.2 自升级防线

AI 在 T2 及以下档位执行时，若变更影响触及项目语义包中标记为 `CRITICAL` 或 `HIGH_RISK` 的实体，**无条件升级至 T3**，内部记录 `[自升级] 命中实体：[实体名]`。无需人类确认。

### 3.3 自动档位推断

若人类未明确指定档位，AI 根据任务描述自动推断：
- 关键词“修复”“微调”“小改”→ T1
- 关键词“新增模块”“重构部分系统”“优化性能”→ T2
- 关键词“底层重构”“接口变更”“存档结构修改”“跨模块”→ T3

推断结果在 ANALYZING 状态向人类确认。

## 四、规则优先级仲裁

### 4.1 优先级定义

| 优先级 | 类别 | 覆盖关系 |
|--------|------|----------|
| P0 | 编译与运行时安全 | 可覆盖一切，任何档位下不可违反 |
| P1 | 数据与资源持久化安全 | 仅 P0 可覆盖 |
| P2 | 架构一致性 | 高于流程规则，低于 P0/P1 |
| P3 | 协作流程（档位行为、文档更新） | 可被 P0-P2 暂时覆盖 |
| P4 | 文档风格与注释 | 最低，紧急情况可暂缓 |

### 4.2 同级冲突仲裁

当两条同优先级规则冲突时，按以下顺序裁定：

| 裁决顺序 | 原则 | 说明 |
|---------|------|------|
| 1 | 安全优先 | 防崩溃 > 防数据损坏 > 防架构污染 > 防性能退化 |
| 2 | 作用域匹配优先 | 精确匹配当前作用域的规则优先于通用规则 |
| 3 | 实体锚定优先 | 绑定到具体 Critical Entity 的规则优先于泛化规则 |
| 4 | 阻断型优先 | 禁止/阻断型规则优先于建议/推荐型规则 |

## 五、认知预算与风险快照

### 5.1 认知预算检查点

AI 在状态跃迁时，内部评估当前活跃治理元素数量：

| 元素 | 预算上限 |
|------|---------|
| 活跃规则条款 | ≤ 25 |
| 活跃 Critical Entity | ≤ 8 |
| 当前任务关联命名空间 | ≤ 5 |
| 当前活跃风险域 | ≤ 5 |

若任一超限，AI 必须裁剪 P3/P4 规则，释放不相关检查项。T3 任务须在产出清单末尾标注 `[上下文裁剪]`。

### 5.2 强制重新聚焦

以下任一条件满足时，AI 必须刷新 Runtime Hazard Snapshot 并丢弃无关历史上下文：
- 已修改文件数 > 8
- 状态回跳次数 > 3
- 当前对话轮数 > 15 且未进入 VERIFYING
- 新增命名空间 ≥ 3
- 发生规则冲突

### 5.3 Runtime Hazard Snapshot

AI 在进入 IMPLEMENTING 前，内部生成快照（不输出给用户）：
[Runtime Hazard Snapshot]

已触及实体：[...]

当前危险等级：P0/P1

禁止依赖检查：通过/有警告

高频路径检查：是否涉及 Update

序列化风险：是否触及存档

强制检查项：[...]

允许修改范围：[...]

禁止修改范围：[...]

## 六、熵控制系统

### 6.1 熵增检测指标

| 指标 | 阈值 | 动作 |
|------|------|------|
| 非基础服务类（Manager/Service）数量 | > 3 且跨 ≥3 命名空间 | T2+ 警告 |
| 临时兼容补丁（TODO/if legacy） | 单文件 ≥ 2 处 | T2 警告，T3 要求记录 |
| 模式碎片化（Coroutine + async 混用等） | 核心路径存在 ≥2 种 | T3 建议统一 |
| 单事件订阅者 | > 5 | T2 警告 |
| 修改文件数持续膨胀 | ≥ 5 且存在补丁式修改 | 触发 Entropy Abort 审查 |
| 架构漂移（同功能多套并行实现） | ≥ 2 套 | 触发 [Architecture Drift] |

### 6.2 Entropy Abort

当修改持续扩散、补丁链增长、架构方向失控时，AI 必须：
1. 停止当前修改（不回滚）。
2. 输出简短的 Entropy Abort Report：已修改文件列表、临时补丁位置、建议替代路线。
3. 任务标记为“需人类决策”。

## 七、任务终止门控

在 COMPLETING 状态，必须满足以下全部条件任务才算完成：

1. 编译 0 Error。
2. 至少一次 Play Mode 运行，关键路径无异常（涉及 P1 以上系统）。
3. 无新增 P0/P1 违规。
4. 无 Forbidden Dependency。
5. 无未解决的 Entropy Abort。
6. 验证契约完成（基准第十四节）。
7. 产出清单包含：变更摘要、新增文件、是否触发自升级。

若**连续 3 次 VERIFYING 失败**且原因不收敛，AI 必须终止任务，进入 RISK_REVIEW，输出完整影响分析。

## 八、动态扩展加载协议

### 8.1 默认加载

AI 在任何任务中默认仅加载核心双文件：
- 本文件（协作协议内核）
- `AI_DEV_STANDARDS.md`（工程技术基准）

加载方式遵循认知预算检查点，不因任务类型全量加载所有章节。

### 8.2 领域扩展激活条件

以下领域在**首次进入**时，可加载对应扩展文件（扩展文件默认不存在，由人类按需创建）：

| 领域 | 扩展文件路径 | 激活条件 |
|------|-------------|----------|
| Steamworks | `Docs/governance/extensions/GOVERNANCE_STEAM.md` | 任务涉及 Steam API |
| 多人联网 | `Docs/governance/extensions/GOVERNANCE_MULTIPLAYER.md` | 任务涉及 Netcode/RPC |
| ECS/DOTS | `Docs/governance/extensions/GOVERNANCE_ECS.md` | 任务涉及 Unity.Entities |
| Modding | `Docs/governance/extensions/GOVERNANCE_MODDING.md` | 任务涉及外部资源加载 |
| DLC | `Docs/governance/extensions/GOVERNANCE_DLC.md` | 任务涉及 DLC 内容管理 |

### 8.3 扩展文件格式规范

每个扩展文件须包含：
- 激活条件声明
- 依赖核心版本声明
- 规则锚点域 `EXT`
- 具体规则（每条带锚点 ID）

加载扩展时，AI 须重新评估认知预算，超限时优先裁剪 P3/P4 规则。
