# Subway Surfers 范本参考下的受控 AI 开发计划

## 目标

基于 `Subway_Surfers.md` 作为参考案例，建立一套可稳定进入开发的 AI 辅助流程。该文档不是通用标准，也不能替代当前项目自己的设计文档。正式需求只能来自当前项目设计文档、用户明确确认的补充内容，或经过记录的 `repair_patch.json`。

这里的“控制范围”不是限制用户已经提交的设计，而是限制 AI 不得添加没有来源、关联很弱、或只是为了通过审核而临时发明的内容。只要当前项目设计文档中已经覆盖长线运营、活动、商业化、上线和玩家支持，它们就是授权范围的一部分；如果当前项目设计文档没有写对应系统，则不做，也不能由 AI 仅凭 `Subway_Surfers.md` 自行补成正式需求。

`Subway_Surfers.md` 可以提供参考，但参考对象不是 Subway Surfers 这个项目的已选结果，而是它背后的设计选项体系：

- 可参考其章节覆盖方式。
- 可参考每个设计问题的候选选项集合。
- 可参考 checklist 覆盖维度和验收维度。
- 不可把其中 `[x]` 已选项、主目标、具体系统、数值、运营、商业化和上线策略直接当作当前项目需求。

本计划默认开发当前项目自己的游戏，不复制 Subway Surfers 的官方角色、美术、品牌、关卡主题或受保护素材。

核心原则：运营计划、活动计划、商业化计划、上线支持计划如果在当前项目设计文档中存在，就按系统开发处理。它们和玩法、UI、数值一样，都是可拆分、可实现、可验收的系统。

## 参考文档使用边界

AI 使用 `Subway_Surfers.md` 时必须遵守：

- 只能使用全量候选选项体系、设计问题、覆盖清单和审查维度。
- 不能继承范本文档中的 `[x]` 已选项。
- 不能把范本文档中的系统默认加入当前项目。
- 不能因为范本文档有某个模块或已选项，就要求当前项目也必须有。
- 如果当前项目文档缺少某模块，只能基于选项体系提出“是否需要”的问题，不能自动开发。
- 如果当前项目文档明确写了某模块，即使范本文档没有，也要按当前项目文档开发。
- 所有正式需求必须追溯到当前项目设计文档或用户确认记录。

## 选项体系使用方法

通用流程应把 `Subway_Surfers.md` 拆成两层：

- `option_taxonomy`：设计域、决策章节、设计问题、候选选项、选项代码、验收维度。这一层可以通用。
- `case_selection`：`[x]` 选项、主目标、Subway Surfers 的实例选择。这一层不能通用。

新项目应这样使用：

1. 读取当前项目设计文档。
2. 用 `option_taxonomy` 检查当前文档覆盖了哪些设计问题。
3. 对当前文档已经明确回答的问题，映射到对应选项。
4. 对当前文档没有回答的问题，生成 `open_questions.json`。
5. 对当前文档明确写出的系统，生成系统关联图和资源流转图。
6. 禁止用 `case_selection` 代替当前项目选择。

也就是说，AI 应该问“当前项目在这个选项体系中选择了什么”，而不是说“Subway Surfers 选了什么，所以当前项目也该选什么”。

## 核心问题

当前流水线中，01-03 阶段主要是导入和校验源资料。它能证明治理文件完整，但不能证明业务内容合理。

如果让 AI 在审核失败后自由补内容，会出现两个典型问题：

- 不完整时，AI 会补出很多原设计没有承诺的系统。
- 为了通过评审，AI 会把缺口扩大成弱关联的新玩法、新经济、新活动，导致设计冻结阶段反而变成无来源扩写。

解决办法是把 AI 修补变成受控动作：**当前项目文档内的内容都要被开发计划接住；文档外的内容只能以最小修补形式进入，并且必须有证据和理由**。

## 推荐 AI 角色

这些不一定要是多个独立 agent，也可以是同一个模型按角色分步执行。稳定性来自 schema 和本地校验器，不来自 agent 数量。

| 角色 | 责任 | 禁止事项 |
|---|---|---|
| Design Extractor | 从当前项目设计文档抽取核心玩法、系统、实体、规则、边界、风险 | 不创造新系统 |
| Scope Governor | 判断当前项目内容属于核心玩法、成长经济、内容供给、社交、长线运营、上线支持或非目标 | 不把参考文档内容伪装成当前项目范围 |
| Completeness Reviewer | 检查是否缺少可开发所需信息 | 不直接改设计 |
| Minimal Repair Writer | 只补阻断开发的最小信息 | 不新增玩法分支 |
| Consistency Reviewer | 检查上下游矛盾、术语混乱、重复承诺 | 不引入替代设计 |
| Program Translator | 把冻结设计转成程序需求、数据结构、状态、事件、验收标准 | 不改变设计冻结内容 |
| Traceability Validator | 检查每个需求是否可追溯到设计来源或批准修补项 | 不接受无来源需求 |

## 受控修补规则

AI 每次修补必须生成 `repair_patch.json`，并满足以下约束：

```json
{
  "issue_id": "GAP-001",
  "patch_type": "clarify | choose | bound | split | defer | remove | map",
  "source_evidence": ["current_design_doc.md:line"],
  "original_gap": "缺口描述",
  "minimal_patch": "最小修补内容",
  "scope_impact": "none | core_play | progression | economy | content | social | live_ops | launch_ops | deferred",
  "complexity_delta": 0,
  "new_mechanics_added": [],
  "requires_human_decision": false
}
```

允许的修补类型：

- `clarify`：把含糊描述改成可开发规则。
- `choose`：在文档已有选项中选择一个。
- `bound`：给已有系统增加边界。
- `split`：把过大的需求拆成多个开发批次，但仍保留在完整产品范围内。
- `defer`：记录为后续批次，不代表不开发。
- `remove`：删除越界补充。
- `map`：建立设计项到程序需求的映射。

禁止的修补：

- 新增当前项目设计文档没有承诺的新玩法。
- 因为某项不完整而扩展成新系统。
- 把当前项目文档没有覆盖、且无法由已有设计强关联推出的内容塞进当前范围。
- 用“增强体验”“丰富内容”“提高留存”作为无边界新增理由。

## 阶段流程

### 00 设计文档导入

输入：当前项目设计文档。`Subway_Surfers.md` 只作为参考案例使用。

输出：

- `design_source_manifest.json`
- `design_extraction.json`
- `full_scope_catalog.json`
- `implementation_phase_map.json`
- `source_digest.md`
- `open_questions.json`

门禁：

- 文档可读取。
- 章节结构可识别。
- 已抽取核心体验、目标用户、范围边界、核心玩法、UI、数值、内容、经济、社交、活动、上线、风险。
- 已把文档内容分入完整范围目录，而不是只保留一个小范围切片。

### 01 玩法框架确认

输出：

- `gameplay_framework.md`
- `core_loop.json`
- `system_boundary.json`
- `full_feature_scope.json`
- `core_playable_slice.json`
- `implementation_phase_map.json`
- `system_relation_graph.json`
- `system_relation_graph.mmd`
- `resource_flow_graph.json`
- `resource_flow_graph.mmd`

核心可玩切片建议：

- 三道移动。
- 跳跃、下滑、左右换道。
- 自动前进和速度递增。
- 障碍物、金币、得分、距离。
- 碰撞失败、结算、重试。
- 2-3 个基础道具，例如磁吸、护盾、加速。
- 一个基础角色和少量可替换外观。

完整产品范围还必须接住：

- 长线赛季。
- 复杂活动系统。
- 付费商品体系。
- 好友和排行榜。
- 多区域运营和客服体系。
- 复杂成长数值。

这些内容不应该被删除。它们应进入 `implementation_phase_map.json`，作为后续开发批次、运营批次或上线准备批次。

步骤 1 必须基于当前项目已选出的系统生成两张图：

- `system_relation_graph`：系统之间的依赖、触发、读取、写入和配置关系。
- `resource_flow_graph`：当前项目资源如何产生、流转、消耗和沉淀。

图中的每个节点都必须来自当前项目设计文档或已批准的 `repair_patch.json`。如果 AI 想新增一个节点，必须先证明来源和必要性；否则只能写入 `open_questions.json`，不能进入图。

### 02 设计评审冻结

该阶段只做冻结，不做创作。

输出：

- `frozen_game_design.md`
- `design_freeze_contract.json`
- `scope_decisions.json`
- `phase_backlog.json`
- `design_review_report.json`

冻结规则：

- 所有冻结内容必须能追溯到当前项目设计文档、用户确认记录或 `repair_patch.json`。
- 所有设计内容必须被分入开发批次，不能因为暂时不做核心可玩切片就丢失。
- AI 发现缺口时，先标记 `blocking | warning | deferred`。
- 只有 `blocking` 可以触发最小修补。
- 修补后必须重新跑一致性审核。
- 连续 3 次仍不通过时，不继续扩写，改为输出阻断问题。

通过条件：

- 阻断项为 0。
- 每个核心可玩切片玩法都有明确输入、反馈、失败条件、验收方式。
- 长线运营、活动、商业化、上线支持等完整范围内容都有开发批次和依赖关系。
- 没有新增未追溯到源文档或修补记录的系统。

### 03 程序需求确认

输出：

- `systems.md`
- `entities.md`
- `contracts.md`
- `acceptance_criteria.md`
- `program_structure_spec.md`
- `program_requirements.md`
- `program_requirements_contract.json`
- `traceability_matrix.json`

程序需求必须覆盖：

- Runner 控制器。
- 赛道和三道坐标。
- 输入系统。
- 玩家状态机。
- 障碍生成和碰撞。
- 道具生成和效果。
- 得分、金币、距离。
- 难度曲线。
- 局内 HUD。
- 结算和重试。
- 配置表和测试用例。
- 成长、收集和解锁。
- 经济循环、货币和商品。
- 任务、活动、赛季和版本节奏。
- 社交、排行榜和轻量异步关系。
- 数据埋点、留存指标和测试设计。
- 合规、未成年人保护、隐私和本地化。
- 上线准备、玩家支持、反馈入口和应急边界。

### 04-09 美术、评审和计划

美术需求只根据已冻结完整范围生成，不允许反向新增玩法。

程序计划必须把 03 的需求拆成开发任务：

- 输入和角色控制。
- 跑道/相机/移动。
- 障碍与收集物。
- 生成器。
- 碰撞和失败。
- 道具。
- UI。
- 配置和存档。
- 测试。
- 成长经济。
- 任务活动。
- 社交排行。
- 数据指标。
- 上线和运营支持。

### 10-12 开发执行与集成

实现顺序：

1. 可运行空场景。
2. 三道移动和输入。
3. 跳跃、下滑、碰撞盒。
4. 障碍物和金币。
5. 速度递增和难度曲线。
6. 得分、距离、金币。
7. 道具效果。
8. HUD、结算、重试。
9. 配置表和自动化测试。
10. 成长、收集、解锁和存档。
11. 任务、活动、赛季和版本配置。
12. 经济、商品、广告或付费相关规则。
13. 排行榜、轻社交和异步关系。
14. 数据埋点、指标看板和灰度实验。
15. 上线支持、客服入口、公告口径和应急流程。

集成验证必须证明：

- 30 秒无异常运行。
- 输入响应稳定。
- 碰撞失败可复现。
- 金币和得分正确。
- 重试不残留旧状态。
- 道具生命周期正确。
- UI 不遮挡核心操作反馈。
- 长线系统不破坏核心跑酷体验。
- 商业化不破坏公平边界。
- 活动和赛季可通过配置关闭或回滚。

## 审核循环

每个 AI 生成阶段使用固定循环：

```text
generate -> review -> classify issues -> minimal repair -> re-review -> gate
```

循环限制：

- 最多 3 轮自动修补。
- 每轮只能处理阻断项。
- 每个阻断项只能产生一个最小 patch。
- patch 不能增加新机制，除非替换掉等量旧机制并说明原因。
- 第 3 轮仍失败时，停止生成，输出 `blocked_issues.json`。

## 本地校验器规则

除了 AI 审核，还需要确定性校验器：

- 检查必须文件是否存在。
- 检查 `imported_sources` 是否不为 0，或者本阶段真实业务内容是否存在。
- 检查所有完整范围需求都在 `full_scope_catalog.json` 或 `repair_patch.json` 中有来源。
- 检查所有需求都有开发批次。
- 检查 `system_relation_graph` 中所有节点都有来源和系统归属。
- 检查 `resource_flow_graph` 中所有资源都有生产者、消费者和存储位置。
- 检查所有程序需求都有 trace。
- 检查所有 trace 指向源文档或 `repair_patch.json`。
- 检查 `blocking_issues` 数量为 0。
- 检查后续批次内容不会被误判为缺失或越界。
- 检查弱关联新增内容没有进入冻结范围。

## 范围和批次预算

为防止 AI 过度补充，不限制用户文档中的完整范围，但限制每个开发批次的承载量：

| 批次 | 目标 | 控制规则 |
|---|---|---|
| Core Playable | 首局可玩、可失败、可重试 | 不接入长线系统依赖 |
| Progression | 收集、解锁、成长、存档 | 不改变核心操作规则 |
| Economy | 货币、奖励、商品、广告或付费 | 不破坏公平边界 |
| Content Ops | 任务、活动、赛季、版本配置 | 必须可配置关闭 |
| Social | 排行榜、好友、异步关系 | 不增加强实时依赖 |
| Launch Ops | 指标、灰度、客服、公告、应急 | 不阻塞核心可玩验证 |

批次只是开发顺序，不是范围裁剪。用户明确要求完整开发时，后续批次必须进入总计划和验收链路。

## 对当前项目的改造建议

1. 把 01-03 从纯导入阶段升级为 AI 生成/审核/修补/冻结阶段。
2. 新增 `ai_review_report.json`、`repair_patch.json`、`traceability_matrix.json`。
3. `validation_report.json` 增加业务质量字段：

```json
{
  "content_exists": true,
  "ai_review_status": "passed",
  "blocking_issues": 0,
  "traceability_valid": true,
  "scope_budget_valid": true
}
```

4. 如果源资料为 0 且没有真实业务内容，阶段状态不能显示为 `success`，应显示 `content_missing`。
5. 阶段 02 的 AI prompt 必须声明：只允许冻结、裁剪、澄清和分批，不允许加入没有来源或弱关联的新设计。

## 判断标准

一个阶段不应该因为“看起来完整”而通过。它必须同时满足：

- 设计来源可追溯。
- 完整设计范围被接住，并分配到明确开发批次。
- 没有未解决阻断项。
- 没有未授权新增玩法。
- 后续批次不会被误删，也不会阻塞核心可玩切片。
- 程序需求和验收标准一一对应。

这样 AI 可以持续帮助审核和修补，但不会把设计越修越大。
